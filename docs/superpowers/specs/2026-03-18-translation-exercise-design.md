# Translation Exercise ‚Äî Design Spec

Date: 2026-03-18

## Overview

A new study exercise type where the AI generates short sentences using a vocab word and the user translates them. An AI judge evaluates the translation and returns structured feedback with scores. Supports both directions: English ‚Üí Chinese and Chinese ‚Üí English.

---

## Backend

### New router: `backend/app/routers/translation_exercise.py`

Prefix: `/api/translation`. Registered in `main.py` alongside existing routers.

Note: `backend/app/services/translation.py` already exists (lesson pipeline translation). The router file is named `translation_exercise.py` to avoid import collision. The API prefix `/api/translation` is still clean and user-facing.

Models are local to the router (following `quiz.py` pattern ‚Äî no additions to `models.py`).

### `POST /api/translation/generate`

Generates sentence pairs using a given vocab word. Called once per translation question during session setup.

**Request:**

`usage` is always sent as a string; empty string when no lesson context is available. The prompt degrades gracefully (generates from general knowledge of the word). `sentence_count` is always 3; it is an explicit field for future flexibility. ### `POST /api/translation/evaluate`

Evaluates the user's translation against a reference. Called on submit inside the exercise component.

**Request:**
```json
{
  "openrouter_api_key": "string",
  "source": "‰ªäÂ§©Â§©Ê∞îÂæàÂ•Ω„ÄÇ",
  "source_language": "chinese",
  "target_language": "english",
  "reference": "The weather is nice today.",
  "user_answer": "Today weather is very good."
}
```

**Response:**
```json
{
  "overall_score": 72,
  "accuracy": { "score": 80, "comment": "Meaning preserved correctly." },
  "grammar": { "score": 60, "comment": "Missing article 'the' before 'weather'." },
  "naturalness": { "score": 75, "comment": "\"Very good\" is understood but \"nice\" sounds more natural here." },
  "tip": "A single actionable improvement tip."
}
```

All scores are integers 0‚Äì100, independently generated by the LLM (not computed averages). The prompt instructs the model to keep `overall_score` holistically consistent with the three sub-scores. Uses Pydantic structured output via OpenRouter.

## Frontend

### Exercise type registration

- `ExerciseMode` union type gains `'translation'`
- `Question` interface in `StudySession.tsx` gains:
  ```ts
  translationData?: {
    sentence: { chinese: string, english: string }  // one sentence, pre-selected at build time
    direction: 'en-to-zh' | 'zh-to-en'
  }
  ```
- `distributeExercises` gains a new `hasOpenRouter: boolean` parameter (added after `hasWriting`). It adds `'translation'` to the available pool when `hasOpenRouter` is true. Call site in `handleStart` passes `Boolean(keys?.openrouterApiKey)`. Since cloze and pronunciation also require OpenRouter, this flag effectively replaces the implicit assumption that was previously unchecked.

### Generation ‚Äî integration into `fetchAIContent` and `handleStart`

Generation follows the exact same build-time prefetch pattern as cloze and pronunciation. `fetchAIContent` is extended to also fetch translation sentences.

**Inside `fetchAIContent`:** Add a third parallel fetch for translation questions.

The existing cloze and pronunciation fetches retain their current behavior: if they fail, the fetch rejects and the outer `catch` in `handleStart` fires (triggering the full fallback). Translation fetches are different: each is per-word and resolves to `null` on failure rather than rejecting ‚Äî so a single failed translation does not trigger the outer catch or affect the cloze/pronunciation fetches.

```ts
const translationEntries = types
  .map((t, i) => t === 'translation' ? pool[i % pool.length] : null)
  .filter(Boolean) as VocabEntry[]

// Inside the existing Promise.all alongside cloze and pron fetches:
const translationResults = await Promise.all(
  translationEntries.map(entry =>
    fetch(`${API_BASE}/api/translation/generate`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        openrouter_api_key: keys?.openrouterApiKey,
        word: entry.word,
        pinyin: entry.pinyin,
        meaning: entry.meaning,
        usage: entry.usage ?? '',
        sentence_count: 3,
      }),
      signal,
    }).then(r => r.ok ? r.json() : Promise.reject())
      .catch(() => null)  // per-item null ‚Äî does NOT affect cloze/pron fetch behavior
  )
)
```

`fetchAIContent` returns `{ clozeExercises, pronExercises, translationResults }` where `translationResults` is an array parallel to translation entries, with `null` for failed fetches.

**Inside `handleStart`, question building loop:** Translation questions with a `null` result are excluded from the final `qs` array (the slot is simply not pushed). This means the total question count may be lower than requested when fetches fail ‚Äî this is consistent with the existing behavior where question count is not guaranteed.

```ts
let translationIdx = 0
const qs: Question[] = []
for (let i = 0; i < types.length; i++) {
  const type = types[i]
  const entry = pool[i % pool.length]
  if (type === 'translation') {
    const result = translationResults[translationIdx++]
    if (!result) continue  // skip failed fetches silently
    const sentences: { chinese: string, english: string }[] = result.sentences
    const sentence = sentences[Math.floor(Math.random() * sentences.length)]
    const direction: 'en-to-zh' | 'zh-to-en' = Math.random() < 0.5 ? 'en-to-zh' : 'zh-to-en'
    qs.push({ type, entry, translationData: { sentence, direction } })
  } else {
    // existing handling for cloze, pronunciation, reconstruction, pinyin, dictation, writing
    const q: Question = { type, entry }
    if (type === 'cloze') q.clozeData = clozeExercises[clozeIdx++]
    if (type === 'pronunciation') q.pronunciationData = pronExercises[pronIdx++]
    if (type === 'reconstruction') q.reconstructionTokens = getReconstructionTokens(entry, entries)
    qs.push(q)
  }
}
```

Note: `handleStart`'s existing `types.map(...)` loop must be refactored into a `for` loop as shown above to support conditional `continue`. The resulting `qs.length` may be less than `count` if translation fetches fail. This is expected ‚Äî `questions.length` (not `count`) is what drives the progress display (`${current + 1} / ${questions.length}`) and session end condition, so this works correctly without further changes.

`'translation'` mode does not get a standalone button in `ModePicker` ‚Äî it appears only in mixed mode. Only mixed-mode distribution assigns translation questions.

**Catch block fallback:** The existing fallback maps `'cloze'` ‚Üí `'pinyin'`. Translation should also fall back to `'pinyin'` (or be dropped from the fallback entirely ‚Äî simplest). Add to the fallback map: `t === 'translation' ? 'pinyin' : t`.

**API base URL:** Use the existing module-level constant `API_BASE = import.meta.env.VITE_API_BASE ?? 'http://localhost:8000'` ‚Äî same as all other fetches in `StudySession`. `openrouterApiKey` comes from `keys?.openrouterApiKey` (same pattern).

### New component: `TranslationExercise.tsx`

Location: `frontend/src/components/study/exercises/TranslationExercise.tsx`

**Props:**
```ts
interface Props {
  sentence: { chinese: string, english: string }
  direction: 'en-to-zh' | 'zh-to-en'
  progress?: string
  onNext: (correct: boolean) => void
}
```

The component calls `/api/translation/evaluate` on submit. `API_BASE` sourced from `import.meta.env.VITE_API_BASE ?? 'http://localhost:8000'` at module level. `openrouterApiKey` sourced from `AuthContext` via `use(AuthContext).keys?.openrouterApiKey` inside the component.

**Phases:**

1. **Prompt** ‚Äî shows source sentence (Chinese for zh-to-en, English for en-to-zh), user types translation:
   - `zh-to-en`: `<input type="text" maxLength={500}>` (plain English)
   - `en-to-zh`: `<ChineseInput>` (existing component with IME/pinyin support); submit button is disabled when the ChineseInput value is empty
   - Submit button triggers evaluate call

2. **Loading** ‚Äî spinner while AI evaluates; submit button disabled

3. **Feedback** ‚Äî structured result card:
   - Overall score badge (color-coded: green ‚â•80, amber ‚â•60, red <60 ‚Äî matching `PronunciationReferee` thresholds)
   - Three rows: Accuracy / Grammar / Naturalness ‚Äî each with score bar + comment
   - Tip block at the bottom
   - "Next" button ‚Üí `onNext(overallScore >= 60)`
   - Amber (60‚Äì79) intentionally counts as correct ‚Äî translation is subjective and partial credit is appropriate

**Error handling (evaluate call):**
- Evaluate fails ‚Üí `toast.error(...)` using `sonner` (same as rest of app), "Next" button appears immediately, `onNext(false)` called

### Rendering in `StudySession`

Add a branch to the session render switch:

```tsx
{q.type === 'translation' && q.translationData && (
  <TranslationExercise
    key={current}
    sentence={q.translationData.sentence}
    direction={q.translationData.direction}
    progress={`${current + 1} / ${questions.length}`}
    onNext={handleNext}
  />
)}
```

`key={current}` ensures the component remounts between questions, consistent with all other exercise branches.

### Session summary

Translation counts toward session summary `correct` count: `overallScore >= 60` = correct, same contract as all other exercises.

## Out of scope

- Retry / revise after feedback (one-shot only)
- Difficulty tiers or hint system
- User-selectable direction preference
- `sentence_count` exposed as a UI setting
